MPC Wallets Explained: Why Enterprises Are Ditching Hardware Wallets

Nov 20, 2024 8 min read PumaPulse Engineering Team
← Back to Blog

Multi-Party Computation (MPC) wallets are replacing hardware wallets and multi-sig setups as the institutional standard for crypto custody. The reason is fundamental: hardware wallets and traditional multi-sig both have single points of failure that MPC eliminates entirely. For exchanges, hedge funds, and custodians managing millions in digital assets, MPC isn't just a better option — it's becoming the baseline expectation.

Key Takeaways

  • MPC splits a private key into shares distributed across multiple parties — the complete key never exists in a single location, eliminating the single point of failure that hardware wallets and traditional custody create.
  • Unlike multi-sig, MPC produces a single standard signature on-chain — no special smart contract support needed, works on any blockchain, and doesn't reveal the signing structure publicly.
  • Threshold signing (e.g., 2-of-3) means the system remains operational even if one party is unavailable or compromised — critical for 24/7 exchange operations.
  • For exchanges and custodians, MPC enables automated signing policies (withdrawal limits, time delays, whitelist checks) that are enforced cryptographically, not just at the application layer.

What is MPC?

Multi-Party Computation is a cryptographic technique that allows multiple parties to jointly compute a function over their private inputs without any party revealing their input to the others. In the context of crypto wallets, the "function" is signing a transaction, and the "private inputs" are key shares. Each party holds a fragment of the private key — a key share — and the signing protocol allows them to collaboratively produce a valid signature without any party ever possessing the complete key.

The most common MPC scheme for crypto wallets is threshold ECDSA (Elliptic Curve Digital Signature Algorithm), which is the signature scheme used by Bitcoin, Ethereum, and most other blockchains. In a 2-of-3 threshold scheme, three key shares are generated and distributed to three parties (or devices). Any two of the three can cooperate to sign a transaction. The third share can be lost, stolen, or its holder can be unavailable — and the wallet remains fully operational. The key is never reconstructed in a single location at any point in the process.

The cryptographic security of MPC is based on the same mathematical hardness assumptions as standard ECDSA — breaking MPC requires breaking the underlying elliptic curve cryptography, which is computationally infeasible with current technology. The additional security comes from the distribution: an attacker must compromise multiple independent parties simultaneously to steal funds, rather than just one device or one person.

MPC vs Multi-Sig

Multi-signature (multi-sig) wallets require multiple on-chain signatures to authorize a transaction. Bitcoin's native multi-sig and Ethereum's Gnosis Safe are the most common implementations. Multi-sig is well-understood, battle-tested, and transparent — the signing structure (e.g., 2-of-3) is visible on-chain, which can be useful for governance and auditability. However, multi-sig has significant limitations that MPC addresses.

The most important difference is on-chain footprint. A multi-sig transaction on Ethereum requires a smart contract (like Gnosis Safe) and includes multiple signatures in the transaction data — making it significantly more expensive than a standard transaction. On Bitcoin, native multi-sig transactions are larger and cost more in fees. MPC produces a single standard ECDSA signature that is indistinguishable from a regular single-key transaction on-chain. This means lower fees, better privacy (the signing structure isn't revealed), and compatibility with any blockchain without requiring smart contract support.

Chain agnosticism is a critical advantage for institutional operators managing assets across dozens of blockchains. A multi-sig setup requires different implementations for each chain — Bitcoin's native multi-sig, Ethereum's Gnosis Safe, Solana's multi-sig programs, and so on. MPC works identically across all ECDSA-based chains (Bitcoin, Ethereum, BNB Chain, Polygon, Avalanche, and hundreds of others) with the same key shares and signing protocol. This dramatically simplifies operations for custodians and exchanges managing multi-chain portfolios.

MPC vs Hardware Wallets

Hardware wallets (Ledger, Trezor, Coldcard) store the complete private key on a dedicated secure element chip. They're excellent for individual users because they isolate the key from internet-connected devices, making remote attacks nearly impossible. But for institutional use, hardware wallets have fundamental limitations. The complete key exists in a single physical device — if that device is lost, stolen, or damaged, the funds are gone (unless you have a backup, which creates its own security risks). Physical access to the device, combined with sophisticated side-channel attacks, has been demonstrated to extract keys from some hardware wallet models.

More importantly, hardware wallets require human interaction for every signing operation. An exchange processing thousands of withdrawals per day cannot have a human physically approve each one on a hardware wallet. MPC enables automated signing with policy enforcement: the signing protocol can be programmed to automatically approve transactions that meet certain criteria (amount below threshold, destination on whitelist, within daily limit) and require human approval for others. This automation is essential for exchange operations while maintaining security guarantees that are stronger than hardware wallets.

Key rotation is another significant advantage of MPC. With a hardware wallet, rotating the key means generating a new key, moving all funds to the new address, and securely destroying the old key — a complex and risky operation. With MPC, key shares can be refreshed (a process called proactive secret sharing) without changing the public key or moving funds. Each party generates a new share that is mathematically related to the old one, and the old shares are destroyed. This means the key material is regularly refreshed, limiting the window of exposure if a share is compromised.

Enterprise Use Cases

Cryptocurrency exchanges are the largest adopters of MPC technology. A production exchange needs to process withdrawals automatically (for user experience) while maintaining security guarantees that satisfy regulators and auditors. MPC enables a tiered signing policy: small withdrawals (under $10,000) are signed automatically by the hot wallet MPC system, medium withdrawals require approval from a risk management system, and large withdrawals require human approval from multiple authorized signers. All of this is enforced at the cryptographic level, not just the application layer — meaning a compromised application server cannot bypass the signing policy.

Hedge funds and asset managers use MPC for treasury management with governance controls. A fund might require 2-of-3 approval from the CIO, CFO, and a designated compliance officer for any transaction above $1M. The MPC system enforces this policy cryptographically — no single person can move large amounts unilaterally, even if they have full access to the fund's systems. This provides both security and regulatory compliance (many jurisdictions require dual-control for large financial transactions).

DeFi protocol treasuries are increasingly using MPC for on-chain governance execution. When a governance vote passes, the treasury multisig (often implemented as MPC rather than Gnosis Safe for the reasons above) executes the approved transaction. The key shares are distributed among core team members and trusted community members, ensuring no single party can act unilaterally while maintaining the ability to execute approved decisions quickly.

Code Example — MPC Signing Policy (Pseudocode)

class MPCSigningPolicy:
    def evaluate(self, tx_request):
        amount = tx_request.amount_usd
        destination = tx_request.destination

        # Whitelist check — always auto-approve
        if destination in self.whitelist:
            return SigningDecision.AUTO_APPROVE

        # Small amounts — auto-approve with rate limiting
        if amount < 10_000 and self.daily_volume + amount < 100_000:
            return SigningDecision.AUTO_APPROVE

        # Medium amounts — require risk system approval
        if amount < 100_000:
            return SigningDecision.REQUIRE_RISK_APPROVAL

        # Large amounts — require 2-of-3 human signers
        return SigningDecision.REQUIRE_HUMAN_QUORUM(2, 3)

Leading MPC Solutions

Fireblocks is the dominant institutional MPC platform, used by hundreds of exchanges, custodians, and financial institutions. It provides a complete custody infrastructure including MPC key management, policy engine, compliance integrations, and a DeFi gateway for interacting with on-chain protocols. Fireblocks charges based on assets under custody and transaction volume — pricing starts at $10,000/month for smaller institutions. Copper and Qredo are strong alternatives with different pricing models and feature sets.

For organizations that need custom MPC implementations — either for cost reasons, specific compliance requirements, or unique operational needs — open-source libraries provide the cryptographic primitives. The tss-lib library (Go, by Binance) implements GG18 and GG20 threshold ECDSA and is used in production by several major exchanges. The multi-party-ecdsa library (Rust, by ZenGo) implements a more recent and efficient protocol. Building on these libraries requires significant cryptographic engineering expertise — the protocols are complex and subtle implementation errors can be catastrophic.
